Summary

Conversation Overview

Steve (affiliated with SUNY Polytechnic Institute and the AIX Center) worked with Claude to design and build a complete podcast knowledge capture and distribution workflow called PodShare. The conversation began with Steve exploring whether Claude could process Spotify podcast links automatically, and evolved through a series of design decisions into a fully specified system with a working skill file. Steve's core problem was wanting to share podcast episodes with different audiences (students, research collaborators, knowledge bases) for different reasons, from a single capture action.

The conversation established that Spotify and Pocket Casts links are not fetchable by Claude, leading to a recommendation of Overcast as the canonical podcast app for this workflow, since Overcast share links resolve cleanly and expose the Apple Podcasts show ID needed to construct pod.link canonical URLs. Steve tested several Overcast links successfully during the session, including episodes from AI Applied (on voice cloning and Box.ai model routing), Science Weekly (on consciousness and AI), and the Google DeepMind Podcast (on Waymo autonomous driving).

The core artifact is the .podshare file format — a flat name-value pair structure with no tags field and no blurb field, consisting of static metadata fields (title, show, host, guest, date, source_link, pod.link, timestamp) and AI-generated dynamic fields mapping episode themes to Steve's projects and knowledge graph. Steve's stated projects include Zettel, LLM-DUN, AIX Workbench, AI Literacy, RTW (Reading-Thinking-Writing), MICA, and Commercial Driver's License. A key design insight Steve articulated was that tags are redundant when everything is a name-value pair, and that the sum of dynamic pairs replaces any need for a blurb field. Steve also noted that user-provided project names should become field names with AI-generated relevance values, and that when his full knowledge graph is available, graph node names should be used as field names automatically.

The workflow produces a .podshare as the content layer, which can then be transformed via natural language commands ("podshare to Zotero," "podshare to Tiddler," "podshare to HTML," etc.) into output formats including TiddlyWiki .tid files, Zotero RIS entries (with the user's "why" mapped to the KW keywords field), self-contained HTML for LMS upload to Brightspace/D2L, Obsidian YAML frontmatter markdown, Twitter threads, and Wikiversity wikitext. Steve named a separate bibliographic tool "Podsight" for the Zotero-specific output stream. The terminology "zed" or "zettel" was established as the software-agnostic semantic object, with "tiddler" being only the TiddlyWiki serialization of it.

Two versions of a blog post were generated during the session — first in Steve's voice, then rewritten in Claude's voice with Steve referred to as "Steve" throughout, framing the workflow as a collaborative design between Steve and Claude. The blog post embeds the full podshare-system-prompt-v2.md skill file for readers to copy and use directly. The canonical skill file was saved to /mnt/skills/user/podshare/SKILL.md during the session. Steve's preferred input format going forward is simply an Overcast URL plus optional why statement or project names — no podcast name required since Overcast links resolve to full metadata.

Tool Knowledge

Web fetch works reliably on overcast.fm URLs, returning episode title, show name, date, and Apple Podcasts show ID in the page metadata. The Apple show ID from Overcast pages constructs pod.link show-level URLs as https://pod.link/[apple_show_id]. Episode-level pod.link URLs require the Apple episode ID (the ?i= parameter from Apple Podcasts URLs), which Overcast share links do not directly expose — the current workaround is to use show-level pod.link as the canonical link. Spotify episode pages, Pocket Casts episode pages, and Apple Podcasts episode pages all block fetching and return no useful metadata. Web search using the show name plus episode topic keywords reliably identifies episodes when direct fetching fails. The skill file was written successfully to /mnt/skills/user/podshare/SKILL.md using mkdir -p followed by cp from the outputs directory.
